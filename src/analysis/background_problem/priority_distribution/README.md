# Priority Distribution 分析モジュール

レビュー優先順位の **「絶対値（分布）」** を可視化・定量化する分析モジュールです。
「優先順位は相対的な順位だけでなく、絶対値としてどう分布しているのか？」という
背景の課題に答えるため、**レビューワーがどれだけ早く対応したか** を代理指標として
リリースサイクルに沿って分析します。

> 設計の詳細・意思決定の経緯は [`design.md`](./design.md) を参照してください。

## 🎯 何を測るか

リリースサイクル内で **新しい Change が投稿されるたび**（＝計測点 `T`）に、
その時点で Open（未マージ・未放棄）な Change を集め、次の所要時間の分布を出します。

| 指標 (metric) | 縦軸の値の意味 |
|---|---|
| `time_to_next_review` | 計測点 `T` から、次の人間レビューコメントまでの時間 |
| `time_to_decision` | 計測点 `T` から、マージ/リジェクト判断までの時間 |

- **横軸**: リリースサイクル内の相対位置（0=サイクル開始, 1=リリース日）
- **縦軸**: 上記の所要時間（既定は対数軸・単位 hours）
- **中心線**: アクティブな Change の所要時間の平均
- **バンド**: 平均 ± (1/2)×標準偏差

## 📁 ファイル構成

| ファイル | 説明 |
|---------|------|
| `main.py` | エントリポイント。全プロジェクト×全メトリクスを実行 |
| `utils/constants.py` | **設定の入口**（対象プロジェクト・リリース・各種パラメータ） |
| `utils/data_loader.py` | ボット名の読み込み・人間/本人判定（Change/リリース日の読込は `common.data_loader` を再エクスポート） |
| `utils/time_utils.py` | `common.time_utils` の再エクスポート（後方互換） |
| `../common/data_loader.py` | **共通**: Change / リリース日の読み込み・サイクル算出 |
| `../common/time_utils.py` | **共通**: 日時パース・所要時間の単位変換・横軸の相対位置計算 |
| `metrics/duration_calculator.py` | メトリクス定義の本体（`decision_fn` / `screening_fn` / `value_fn`） |
| `metrics/distribution_builder.py` | 計測点ごとの平均±std バンドの時系列を構築 |
| `preprocessing/outlier_filter.py` | 未決/放置 Change の除外＋両側 IQR による外れ値除去 |
| `io/result_writer.py` | 分布データの csv / json 出力 |
| `visualization/plotter.py` | バンド付き折れ線グラフの描画（日本語フォント自動選択） |

## 🚀 実行方法

プロジェクトのルートディレクトリ（`src/` がある場所）で実行します。

```bash
python -m src.analysis.background_problem.priority_distribution.main
```

> `-m` はモジュールをスクリプトとして実行するオプションです。`from src.~~~` の
> 絶対インポートを解決するため、ファイルパス（`main.py`）ではなくモジュール名で起動します。

実行すると `utils/constants.py` の `TARGET_PROJECTS` × `ENABLED_METRICS` をすべて処理します。
（nova は約 4 万件の Change を読むため、初回ロードに数十秒〜数分かかります。）

## ⚙️ 設定の変え方（`utils/constants.py`）

主要なパラメータはすべてここに集約しています。

| 設定 | 説明 |
|---|---|
| `TARGET_PROJECTS` | 分析するプロジェクトと、そのリリース version の一覧 |
| `ENABLED_METRICS` | 実行する縦軸メトリクス（`time_to_next_review` / `time_to_decision`） |
| `OUTLIER_METHOD` / `OUTLIER_IQR_K` | 外れ値除去の方法と IQR 係数（両側） |
| `X_AXIS_MODE` | 横軸（`normalized` / `days_until_release`） |
| `BAND_STD_FACTOR` | バンド幅の係数（既定 0.5 = 平均±0.5std） |
| `Y_LOG_SCALE` / `DURATION_UNIT` | 縦軸の対数軸 ON/OFF、所要時間の単位（hours / days） |
| `MIN_ACTIVE_FOR_POINT` | 値が得られた Change がこの数未満の計測点は出力しない |
| `MIXED_PLOT_BINS` | 全リリース混合図を等幅ビン平均にする場合のビン数（None=全点） |

### 縦軸メトリクスの追加
新しい縦軸変数を増やしたい場合は、`metrics/duration_calculator.py` に
3 つの関数（`decision_fn` / `screening_fn` / `value_fn`）を定義して
`METRIC_REGISTRY` に登録し、`constants.ENABLED_METRICS` に名前を足すだけです。

## 📤 出力

`data/analysis/background_problem/priority_distribution/` 配下に保存されます。

```
<project>/
├── <version>/                 # リリースごと（例: 20.0.0）
│   ├── time_to_next_review/
│   │   ├── distribution.csv
│   │   ├── distribution.json
│   │   └── distribution.png
│   └── time_to_decision/
│       └── ...
└── all_releases/              # 全リリースを混ぜたもの
    ├── time_to_next_review/
    └── time_to_decision/
```

### CSV（`distribution.csv`）

| 列 | 説明 |
|---|---|
| `x` | 横軸の相対位置 |
| `timestamp` | 計測点 T の実時刻（ISO 文字列） |
| `n_active` | その計測点で値が得られた Change 数 |
| `mean` / `std` | 値の平均・標準偏差 |
| `lower` / `upper` | バンド下端・上端（`mean ∓ 0.5*std`） |
| `release_version` | （混合図のみ）その点が属するリリース |

### JSON（`distribution.json`）
`meta`（分析条件・対象/除外件数・外れ値設定 など）と `points`（データ点）を含み、
図の再現に必要な文脈を保存します。

## 📝 主な定義・前提（詳細は design.md）

- **アクティブ集合**: `created <= T < decision_time`（Gerrit の Open 状態に統一）
- **decision_time**: status が MERGED / ABANDONED の Change の `updated` を採用
- **人間のレビュー**: 次を除外して残ったコメント。投稿者本人 ／ 自動生成メッセージ
  （`tag` が `autogenerated:` で始まる）／ ボット・自動アカウント。
  ボット判定は **3 つの一覧の和集合**（すべて大文字小文字無視で `name`/`email`/`username` を照合）:
  1. `src/config/gerrymanderconfig.ini` の `[organization] bots`（username）
  2. `src/config/third_party_ci_accounts.csv`（サードパーティ CI の `name` / `email`）
  3. `src/config/extra_bots.txt`（`jenkins` / `zuul`。上記未登録の補完。1 行 1 名）
- **放置/外れ値除去**: 未決や「一度もレビューされない」Change を除外し、
  さらに代表所要時間に対して両側 IQR で外れ値を除外

## ⚠️ 結果を解釈・記述するときの注意

データ処理には、見落としやすい **暗黙のフィルタ条件** がいくつかあります。
結果を読むとき・他者に説明するとき・論文やスライドに手法を記述するときなど、
**常にこれらの前提を踏まえ、必要に応じて併記し忘れない**よう注意してください。

- **少数データ時の外れ値除去スキップ**: あるリリースの有効 Change 数が
  `MIN_RECORDS_FOR_OUTLIER`（既定 **4 件**）未満の場合、分位点（Q1/Q3）が不安定になるため
  **外れ値除去を行わず全件を残します**（[`preprocessing/outlier_filter.py`](./preprocessing/outlier_filter.py)）。
  → 「両側 IQR で外れ値除去した」とだけ書くと不正確。「ただし○件未満のリリースは除去せず全件保持」も明記する。
- **人間レビューの除外条件**: 上記「人間のレビュー」のとおり、投稿者本人・自動生成メッセージ・
  ボット（gerrymander ＋ CI 一覧 CSV ＋ jenkins/zuul）を除外しています。
  「ボットを除外した」だけでなく、用いたボット一覧の出所も記述する。
  なお現状の一覧は**当時の表記揺れ・新規 CI を完全には網羅しません**（例: `Cloudbase Nova Hyper-V CI`
  等が残存しうる）。厳密化するには CSV への追記、または振る舞いベースの同定への切替を検討する。
- **計測点と対象 Change**: 計測点 `T` は各リリースの開発サイクル `[直前リリース日, 当該リリース日]`
  内に投稿された Change の `created`。各計測点 `T` でのアクティブ集合（対象 Change）は
  **`T - LOOKBACK_DAYS（既定 365 日）<= created <= T < decision_time`**
  （= `T` から 1 年以内に投稿され、かつ `T` 時点で Open）。
  → サイクル開始前に投稿されても、`T` 時点で Open かつ 1 年以内なら対象に含まれる点に注意。
- **未決・放置の除外**: `status` が MERGED/ABANDONED でない Change、および指標の
  screening 値が未定義（例: 一度も人間レビューが無い）Change は分析前に除外しています。

> これらの条件は実行ログ（`drop_unfinished` / `drop_outliers` の件数）と、各出力の
> `distribution.json` の `meta`（`n_changes_total` / `n_dropped_unfinished` / `n_dropped_outliers`）に
> 記録されるので、実際の除外件数はそこで確認できます。
