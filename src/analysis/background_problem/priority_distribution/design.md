# 優先順位の「分布」分析 設計書 (priority_distribution)

## 0. このドキュメントについて

本ドキュメントは、研究の **背景の課題** に対する事前分析コードの設計書です。
実装に入る前に「何を・どう作るか」を確定させることを目的とします。

---

## 1. 背景と目的

### 1.1 研究全体の文脈
- AI の発展により PR の投稿難易度が下がり、PR 投稿数が増加した。
- 投稿数増加により「どの PR を優先的にレビューすべきか」の **優先順位付け** が難しくなった。
- 従来研究はレビュー優先順位を自動付けするモデルを提案したが、期間（機能追加・バグ修正など）に
  応じて優先順位が変動する PR を捉えきれなかった。

### 1.2 この分析が答えたい問い
レビューワーへの指摘:

> 「優先”順位”まで決めてるの？順位という相対値ではなく、**絶対値的にはどうなってる？**
> つまり優先順位の **分布** は？」

つまり「優先順位＝相対的な順位」だけでなく、**絶対値としてどう分布しているか** を可視化・定量化する。

### 1.3 優先順位の「絶対値」の代理指標 (proxy)
優先順位そのものは観測できないため、**レビューワーが実際にどれだけ早く対応したか** を代理指標とする。
レビュー優先度が高い PR ほど対応が早い、という仮定に基づく。代理指標は次の 2 つ:

| 指標名 (metric) | 縦軸の値の意味 | 値の計算 |
|---|---|---|
| `time_to_next_review` | **計測時点 `T` から、次の人間レビューコメントまでにかかった時間** | `(T 以降で最初の人間コメントの時刻) - T` |
| `time_to_decision` | **計測時点 `T` から、導入(マージ)/却下(リジェクト)の判断までにかかる時間** | `decision_time - T` |

- 両指標とも **計測時点 `T` を起点** にした残り時間として定義する。
  すでにレビュー済みで Open 中の Change も、`T` 以降の次レビュー／判断までの時間として寄与する。
- `time_to_next_review` は「Change の最初のレビュー（created 起点）」ではない点に注意（§2.2）。

この 2 指標を **同じ枠組みで処理** し、縦軸の変数を柔軟に切り替えられるようにする（§5.3）。

## 2. 用語と定義（厳密化）

実装のブレを防ぐため、曖昧になりがちな点を先に固定する。

### 2.1 Change（=PR）
本データは OpenStack / Gerrit のため PR を **Change** と呼ぶ。
`data/openstack_collected/<project>/changes/*.json` に 1 Change = 1 ファイルで保存されている。

主な利用フィールド:
- `created` … 投稿時刻（文字列 `"YYYY-MM-DD HH:MM:SS.fffffffff"`）
- `updated` … 最終更新時刻（同形式）
- `status` … `MERGED` / `ABANDONED` / `NEW` など
- `owner.email`, `owner.name`, `owner.username` … 投稿者
- `messages[]` … 時系列イベント。各要素に `author{email,name,username}`, `date`, `message`, `tag`

### 2.2 人間のレビューコメント / 「次のレビュー」
**人間のレビューコメント** = ボットと投稿者本人を除いたコメント。

人間判定（次をすべて除外）:
- 自動生成メッセージ（Gerrit の `tag` が `autogenerated:` で始まる。例: `autogenerated:zuul:check`）。
- ボット・自動アカウント（次の 3 一覧の和集合に一致、§3.3。`name`/`email`/`username` を
  大文字小文字無視で照合）:
  1. `gerrymanderconfig.ini` の `[organization] bots`（username）
  2. `third_party_ci_accounts.csv`（サードパーティ CI の name / email）
  3. `src/config/extra_bots.txt`（`jenkins` / `zuul`）
- 投稿者本人（`author` が `change.owner` と一致：email/username/`_account_id` のいずれか）。

> 注: この一覧方式は当時の表記揺れ・新規 CI を完全網羅しないため、`Cloudbase Nova Hyper-V CI` 等の
> 取りこぼしが残りうる。将来は「Verified 投票のみ／ビルド結果メッセージのみ」のような
> 振る舞いベースの同定への切替を検討する（その場合 §3.3 を更新）。

**「次のレビュー」（計測時点 `T` 基準）**:
- ある計測時点 `T` における Change の「次のレビュー」とは、
  `messages` を `date` 昇順に見て **`date > T` を満たす最初の人間コメント** を指す。
- 縦軸の値 = `その人間コメントの時刻 - T`。
- `T` 以降に人間コメントが存在しない Change は、その計測点では **寄与しない**（スキップ）。

> 「最初のレビュー」ではなく「**次のレビュー**」である点が要点。
> 計測時点で既にレビュー済みの Open な Change も、`T` 以降に来る次レビューまでの時間として扱う。
> 新規投稿された Change（`created == T`）の場合は「次のレビュー＝最初のレビュー」となる。

### 2.3 マージ/リジェクト判断時刻（decision time）
- `status == "MERGED"` または `status == "ABANDONED"` の Change のみ判断済みとみなす。
- 判断時刻は次の優先順で決定（実装は `duration_calculator` に集約、設定で切替可）:
  1. 終局を示す `messages`（merge/abandon を示すメッセージ）の `date` が取れればそれ。
  2. 取れなければ `updated` を判断時刻の近似として用いる（既定）。
- `status` がそれ以外（`NEW` 等、まだ未決）なら **対象外**（§6 で除外）。

### 2.4 計測時点（measurement event）とアクティブ集合
- **新たな Change が投稿された瞬間** を 1 つの計測点とする（その Change の `created`）。
  計測点はリリースサイクル `[cycle_start, cycle_end]` 内に投稿された Change の `created`。
- ある計測時点 `T` における **アクティブ集合** は、**両指標とも共通** で

  ```
  T - LOOKBACK_DAYS <= created <= T < decision_time
  （T から LOOKBACK_DAYS（既定 365 日＝1 年）以内に投稿され、かつ T 時点で Open）
  ```

  を満たす Change 全体（新規投稿された Change 自身も含む）。
- すなわちアクティブ判定は **「T から 1 年以内に投稿」＋「Gerrit の Open 状態（未マージ・未放棄）」**。
  サイクル開始前に投稿された Change でも、`T` 時点で Open かつ 1 年以内なら含む
  （候補プールは `[cycle_start - LOOKBACK_DAYS, cycle_end]` から読み込む）。
  `time_to_next_review` も「最初のレビュー待ち」ではなく「上記条件を満たす限りアクティブ」とする。

### 2.5 縦軸の値（y）
アクティブ集合の各 Change が計測点 `T` で寄与する値は、指標ごとに次で定義する
（いずれも全履歴から一意に確定する）:

- `time_to_next_review`: `(T 以降で最初の人間コメントの時刻) - T`（§2.2）。
  `T` 以降に人間コメントが無い Change は寄与しない。
- `time_to_decision`: `decision_time - T`（計測時点 `T` から、マージ/リジェクト判断までの残り時間）。

計測点 `T` ごとに、アクティブ集合のうち寄与する値から **2 種類の要約**を計算する:

**(A) 平均±標準偏差**
- 中心線 = 値の **平均**
- バンド = **平均 ± (1/2)×標準偏差**（`lower = mean - 0.5*std`, `upper = mean + 0.5*std`）
- n=1 のとき std=0（バンド幅 0）。

**(B) 分位点（percentiles）**
- 値を昇順に並べ、`PERCENTILES`（既定 `[10, 30, 50, 70, 90]`）に当たる値を `np.percentile` で求める。
- 平均は右に裾を引く分布で大きい側へ引っ張られるため、**外れ値にロバストな中央値(50%)・分位点**で
  分布の形を表すのが狙い（§5.7 で 5 本の折れ線として描画）。

**計測点を出力する最小件数**:
寄与する Change 数 `n_active` が **`max(MIN_ACTIVE_FOR_POINT, len(PERCENTILES))` 未満**の計測点は
出力しない（分位点の本数＝線の本数分のデータが無いと各分位線を意味のある値として定義できないため）。
例えば `PERCENTILES=[10,30,50,70,90]`（5 本）なら、`n_active < 5` の計測点はスキップする。
データ（csv/json）と図（mean±std / percentiles）は同じ計測点集合を共有するため、この判定は両図に適用される。

### 2.6 横軸の値（x） … リリースまでの相対位置
- 各リリース `R` の **開発サイクル** を `[prev_release_date, R_release_date]` と定義する
  （`prev_release_date` は同プロジェクトの 1 つ前のメジャーリリース日）。
- 計測点 `T` の横軸位置は **サイクル内の相対位置（正規化進捗）**:

  ```
  x = (T - cycle_start) / (cycle_end - cycle_start)   # 0.0 = サイクル開始, 1.0 = リリース日
  ```

- **等間隔ではなく、実時刻に基づく相対位置** をそのまま x 値として打点する（依頼要件）。
- リリースを跨いで混ぜる図（§7）では、リリースごとに長さが違うため
  **正規化進捗 [0,1] を用いることで横軸を比較可能** にする（既定）。
- 設定で `days_until_release`（= `cycle_end - T` の日数、軸反転）も選べるようにする。

### 5.3 `metrics/duration_calculator.py` … 縦軸変数の柔軟化の中心
各メトリクスを **3 つの関数の組** として定義し、レジストリに登録する。
縦軸変数の追加・変更はこのファイルへの関数追加だけで完結する。

```python
@dataclass
class MetricDefinition:
    name: str
    # アクティブ判定に使う「Change が閉じる時刻」(= decision_time)。両指標共通。None なら未決。
    decision_fn: Callable[[dict], datetime | None]
    # 外れ値判定・放置除外に使う「Change 1 件あたりの代表所要時間」。None なら放置として除外。
    screening_fn: Callable[[dict, set[str]], timedelta | None]
    # 計測点 T における縦軸値（寄与しないなら None）。
    value_fn: Callable[[dict, datetime, set[str]], timedelta | None]

# 共通: decision_time（§2.3, updated を採用）
def decision_time(change) -> datetime | None: ...
# 人間コメント列（ボット・本人除外, date 昇順）
def human_comment_times(change, bot_names) -> list[datetime]: ...

METRIC_REGISTRY = {
    "time_to_next_review": MetricDefinition(
        name="time_to_next_review",
        decision_fn=decision_time,
        # screening: 最初の人間レビューまでの待ち時間（レビュー無し=None→除外）
        screening_fn=lambda c, b: first_human_review(c, b) - created(c) if first_human_review(c, b) else None,
        # value: T 以降の最初の人間コメント - T（無ければ None）
        value_fn=lambda c, T, b: next_human_review_after(c, T, b) - T if next_human_review_after(c, T, b) else None,
    ),
    "time_to_decision": MetricDefinition(
        name="time_to_decision",
        decision_fn=decision_time,
        # screening: 放置判定用の代表値＝投稿起点の全所要時間（T 非依存の per-Change スカラ）
        screening_fn=lambda c, b: decision_time(c) - created(c) if decision_time(c) else None,
        # value: 計測時点 T から判断までの残り時間（T 依存）
        value_fn=lambda c, T, b: decision_time(c) - T if decision_time(c) else None,
    ),
}

def compute_change_records(changes, metric, bot_names) -> list[ChangeRecord]:
    """各 change について created / decision_time / screening_duration を確定させて返す。"""
```

`ChangeRecord`（中間表現）: `change, change_number, created, decision_time, screening_duration`。
- アクティブ判定（§2.4）は `created <= T < decision_time`（両指標共通）。
- `screening_duration` は外れ値・放置除外（§5.4）に使う代表スカラ。
- 計測点ごとの縦軸値は `distribution_builder` 側で `metric.value_fn(change, T, bot_names)` を呼んで得る。

### 5.4 `preprocessing/outlier_filter.py`
外れ値判定は各 Change の `screening_duration`（§5.3）に対して行う。

| 関数 | 役割 |
|---|---|
| `drop_unfinished(records) -> list` | `decision_time is None`（未決）または `screening_duration is None`（放置＝レビュー無し等）の Change を除外 |
| `drop_outliers(records, method, **params) -> (kept, dropped)` | `screening_duration` に対し **両側** で外れ値を除外 |

両側 IQR:
```
Q1, Q3 = quartiles(screening_durations)
IQR = Q3 - Q1
下側フェンス = Q1 - K*IQR,  上側フェンス = Q3 + K*IQR
[下側フェンス, 上側フェンス] の外を除外（上側＝放置, 下側＝異常に速い）
```

> 「ずっと放置されている Change が分析の邪魔にならないように」という要件は
> ① 未決/放置（screening 未定義）の除外、② screening 値の両側外れ値除外、の 2 段で担保する。
> 除外件数はログと JSON のメタ情報に残し、再現性を確保する。

> ⚠️ **少数データ時のスキップ**: リリース内の有効レコード数が
> `MIN_RECORDS_FOR_OUTLIER`（既定 4 件）未満の場合、分位点が不安定になるため
> ②の外れ値除去を行わず全件を残す。

### 5.5 `metrics/distribution_builder.py` … 時系列構築
```python
def build_distribution(records, metric, bot_names, cycle_start, cycle_end,
                       x_mode, duration_unit, min_active,
                       percentiles=[10, 30, 50, 70, 90]) -> list[DistributionPoint]:
    """
    1. records の created のうち、サイクル内のものを計測点 T とする。
    2. 各計測点 T でアクティブ集合（T - LOOKBACK_DAYS <= created <= T < decision_time）を抽出。
    3. アクティブな各 Change について metric.value_fn(change, T, bot_names) を計算し、
       None でない値（指定単位に変換）だけ集める。
    4. 集めた値から mean / std / n と、各 percentile 値（np.percentile）を算出。
    5. x = relative_x(T, ...) を付与。
    返り値: DistributionPoint(
        x, t, n_active,
        mean, std, lower = mean - 0.5*std, upper = mean + 0.5*std,
        percentiles = {10: v10, 30: v30, 50: v50, 70: v70, 90: v90},
    )
    """
```
- 1 計測点につき **平均±std（A）** と **分位点（B）** の両方を保持する（§2.5）。
- 分位点は `percentiles`（既定 `[10,30,50,70,90]`、`constants.PERCENTILES`）で本数・値を変更可能。
- `n_active` は「その計測点で実際に値が得られた Change 数」。
  **`max(min_active, len(percentiles))` 未満**の計測点は出力しない（分位線を定義できないため。§2.5）。
- 計測点はサイクル内に投稿された Change の `created` のみ。アクティブ集合は §2.4 のとおり
  `T - LOOKBACK_DAYS <= created <= T < decision_time`（サイクル開始前の投稿も含みうる）。
- 全リリース混合（§7）では全リリース分の `DistributionPoint` を x（正規化進捗）で束ね、
  必要なら等幅ビン平均（任意・設定）でならす。既定は全点プロット（添付画像のノイジーな見た目に一致）。

### 5.6 `io/result_writer.py`
- `write_csv(points, path)` / `write_json(points, meta, path)`。
- JSON にはデータ点に加え **メタ情報**（project, version, metric, cycle 期間,
  対象/除外件数, 外れ値設定, x/y 設定）を含め、図の再現に必要な文脈を保存する。

### 5.7 `visualization/plotter.py`
`DistributionPoint` 列から **2 種類の図**を描く。縦軸は所要時間（`Y_LOG_SCALE=True` なら対数）、
横軸は相対位置（0→1 でリリースに近づく）。いずれもリリース別・全リリース混合で共通の関数を使う。
対数軸対策として、0 以下になり得る値は下限を微小正値でクリップする。

- `plot_band(points, ...) -> distribution_mean_std.png`
  - `ax.plot(x, mean)` … 濃い中心線
  - `ax.fill_between(x, lower, upper, alpha=0.3)` … 平均±0.5std の薄いバンド
- `plot_percentiles(points, percentiles, ...) -> distribution_percentiles.png`
  - 各 percentile（既定 10/30/50/70/90）について `ax.plot(x, [p.percentiles[q] for p in points])`
    で **1 本ずつ折れ線**を描く（計 5 本、色違い・凡例つき。バンドは描かない）。
  - 平均が裾に引っ張られる問題を避け、中央値(50%)・分位点で分布の位置と広がりを表す。

## 7. 全リリース混合図の扱い
- 各リリースで `created` を **そのリリースサイクル内の正規化進捗 [0,1]** に変換済みなので、
  全リリースの `DistributionPoint` を x で重ねられる。

- 任意機能として等幅ビン（例: 0.0–1.0 を 50 分割）での平均±std 集約も `constants.py` で
  切替できるようにする（見やすさ優先のとき用）。

## 8. エッジケース・注意点
- **タイムゾーン**: Gerrit は naive datetime。比較時は naive で統一（既存実装と同じ）。
- **n=1 の計測点**: std=0 として出力（バンド幅 0）。`MIN_ACTIVE_FOR_POINT` 未満は出力しない。
- **`time_to_next_review` で `T` 以降に人間コメントが無い Change**: その計測点では寄与しない（§2.2）。
- **対数軸と lower<=0**: 対数描画時は下限を微小正値にクリップ。
- **リリースが TARGET に無い / 1つ前が無い**: 該当リリースをスキップしログに記録。
- **リリース日の異常エントリ**: nova の `2010.1` / `2011.1` は `release_date` が 2018-02-21 と
  実際の版数順序に矛盾する（17.0.0=2018-02-28 の直前に誤って並ぶ）。`get_release_cycle` の
  「直前リリース」は **版数順と日付順が整合するエントリ** から選び、こうした古い版の遅延エントリは
  `data_loader` で除外する（cycle_start を取り違えないため）。
- **JSON が list / 単一 dict 両形式**: `load_changes` で吸収。
- **本人コメント・ボットの除外漏れ**: `is_owner` は email→username→_account_id の順で照合。

## 11. 確定事項（レビュー反映済み）
1. **decision time の決定方法**: `updated` を採用する（§2.3）。
2. **アクティブ集合の定義**: 両指標とも `created <= T < decision_time` に統一＝
   実質的に Gerrit の Open（未マージ・未放棄）準拠（§2.4）。
3. **対象リリース**: nova の `2015.1.0`, `12.0.0`〜`20.0.0`（§4 `TARGET_PROJECTS`）。
4. **所要時間の単位**: hours（§4 `DURATION_UNIT`）。
5. **縦軸 `time_to_next_review` の定義**: 「Change の最初のレビュー」ではなく
   **計測時点 `T` から次の人間レビューまでの時間**（§1.3, §2.2, §2.5）。
6. **バンド幅**: 平均 ± (1/2)×標準偏差（§2.5）。
7. **外れ値除去**: screening 値に対する **両側 IQR**（`Q1-K*IQR`〜`Q3+K*IQR`）（§5.4）。
