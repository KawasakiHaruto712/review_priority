# 変化区間の存在確認 設計書 (concept_drift_existence)

## 0. このドキュメントについて

本ドキュメントは、研究の **目的（レビュー優先順位の判断基準が変化する区間の特定）** に向けた
**事前分析①「変化区間は本当に存在するのか」** のコード設計書です。
実装に入る前に「何を・どう作るか」を確定させることを目的とします。

設計の根拠（検討経緯・なぜその案にしたか・落とした案）は `research_design_notes.md`
（手描き図5枚: 手法 / 変化区間 / 計測時点 / 層分割 / SHAP に対応）にあります。本書はその結論を
実装可能な形に落としたものです。判断に迷ったら notes 側を参照してください。

> **初学者が読み書きできることを最優先**にします。具体的には:
> - 1 ファイル 1 責務。1 関数は短く（目安 5〜30 行）、賢い抽象より素直な手続きを優先。
> - モデルや評価は scikit-learn / LightGBM の高レベル API をそのまま使い、自前実装は最小限。
> - 設定値はすべて `utils/constants.py` の 1 箇所に集約（初学者が最初に触る場所）。
> - 各モジュールの想定行数を §5 の表に明記（肥大化したら分割のサイン）。

## 1. 背景と目的

### 1.1 研究全体の文脈と本分析の位置づけ
- 背景: AI の発展で PR の投稿難易度が下がり、PR 投稿数が増加した。
- 課題: 投稿数増加により「どの PR を優先的にレビューすべきか」の優先順位付けが難しくなった。
- 従来研究: レビュー優先順位を自動付けするモデルを提案したが、期間（機能追加・バグ修正など）に
  応じて優先順位が変動する PR を捉えきれなかった。
- 先行研究（直前）: **期間別のモデル**を作ることで、期間に応じてレビューされやすい/されにくい
  PR を捉えた。ただし **期間別モデルの「切り替えタイミング」は考慮されていない**。
- 本研究の目的: **レビュー優先順位の判断基準が変化する区間（＝切り替えるべき境目）の特定**。
- **本分析（事前分析①）が答える問い**: そもそも **「判断基準が変化する区間」は存在するのか？**
  存在しなければ「期間別に切り替える」という研究の前提が崩れるため、まずこれを検証する。

### 1.2 通底する軸: 概念ドリフト vs データドリフト
本分析で見たいのは **概念ドリフト（判断基準 `P(y|x)` が変わった）** であって、
**データドリフト（入力分布 `P(x)` が変わった）** ではありません。
- 概念ドリフト = 同じ特徴の PR でも、時期によってレビューされやすさ（優先順位）が変わる。
- 設計上のほぼ全ての工夫（距離固定での位置比較 §7、層分割 §2.6）は、この 2 つを切り分けるためのもの。

### 1.3 全体の中での主従関係（重要）
本研究の事前分析は 2 段で構成され、両者は **主従関係** にあります。
- **① 変化区間の存在確認 ＝ 主分析（本書）**。研究の主張はこれ単体で完結する。
- **② 古い PR の SHAP 比較 ＝ 追加実験（補助）**。①で「変化した」と分かった後に *原因* を掘る話。
  **①の結論は②の成否に依存しない**。本書では §9 のとおり **ディレクトリだけ予約** し、設計・実装は後回し。

## 2. 用語と定義（厳密化）

実装のブレを防ぐため、曖昧になりがちな点を先に固定します。

### 2.1 Change（=PR）
本データは OpenStack / Gerrit のため PR を **Change** と呼ぶ。
`data/openstack_collected/<project>/changes/*.json` に保存（読み込みは §3）。

### 2.2 計測時点 `T` とレコード（毎日 0 時の定点スナップショット）
- **計測時点 `T` = 毎日 0:00:00 の定点グリッド**（`MEASUREMENT_STEP_DAYS=1`、刻みは設定可変）。
  「投稿時刻」ではなく規則的な日次グリッドにする（投稿密度による偏りが無く、再現性が高く、毎日のレビュー
  待ち行列を見るという解釈が自然。実装上もレコード数が制御でき現実的）。
- 各計測時点 `T` での **アクティブ集合** = `T` 時点で Open な Change
  （`T - LOOKBACK_DAYS <= created <= T < decision_time`、priority_distribution と同じ考え方）。
- **1 レコード = (Change, T) の組**。アクティブな各 Change は、その日の `T` で 1 レコードになる。
  → **長く Open な Change ほど多くの日次スナップショットに登場し、複数レコードになる**（あと100日→あと70日→…）。
  この **「推移」自体に意味がある**（順位の推移＝優先のされ方の変化を見たい）ので、複数レコードを採用する。
- 特徴量とラベルは **その計測時点 `T` 基準** で計算する（特徴は `T` 時点で観測可能な情報のみ。
  ラベルは §2.3 の `time_to_next_review`＝`T` から次の人間レビューまでの時間）。

> 計測点は「各 Change の投稿時刻（密・不規則）」ではなく **毎日 0 時の定点（規則的）** とする。当初設計の
> 「複数計測点（推移）」の趣旨は保てる（長寿 Change は複数日に登場）。学習も評価も **各 (Change, T) レコードを
> 独立に**扱う（§3.5）。同一 Change が別の `T` で複数回 学習・評価されてよい（クラスタ補正はしない）。
> ただし **同一 Change が学習と評価の両方に出る「またぎ」は禁止**（§2.6 の層2 排他振り分けで防ぐ）。
> 実データでは日次グリッドで 1 リリース約25万・10リリース約240万レコードになるため、developer/project 系の
> 特徴量は**事前集計＋二分探索で高速化**する（定義は `src/features` と同一、同値テストで保証。§3.4, §5.4）。

### 2.3 ラベル `time_to_next_review`（順位の代理、単体で閉じた未来イベント）
- ラベル = `T` から **次の人間レビューコメントまでの時間**
  = `(T 以降で最初の人間コメントの時刻) - T`。
- **人間のレビューコメント** = ボットと投稿者本人を除いたコメント（判定ロジックは
  priority_distribution と同一。§3.3）。
- **未来まで見て付ける（観測窓＝収集済みデータの末尾, 2025年頃まで）**: `T` 以降、**収集している最新
  リリースまで（およそ 2025 年）** の全期間を先読みし、最初の人間コメントが来た時刻でラベルを確定する。
  Change は分割をまたいで重複しない（被らない）ため、評価期間の外まで先読みしてもリークにはならない。
  → 「評価期間内にレビューされなかった Change がラベルを得られず消える」という生存（後ろ向き）
  バイアスを、観測窓を末尾まで広げることで避ける。
- それでも `T` 以降・2025年頃の収集末尾まで人間コメントが一度も無い Change（最後まで未レビュー）は
  **対象外として除外**する（`CENSORING_MODE="drop"`）。**結果に影響を及ぼすほどの件数は無い**想定。
  除外件数はログ・JSON に残す。将来必要なら `survival` に切替可能（§4）。
- **順位はラベルにしない**: ラベルは「その Change 単体で閉じた未来イベント（時間）」とし、他 Change を
  参照しない（リーク回避）。順位は §6・§7 のとおり **評価時に予測スコアを並べて事後的に作る**。

### 2.4 計測時点トリガー `X` と母集団（分割内で閉じる）
- 計測時点 `T` は「ある Change が投稿された瞬間」。この **時点を切るトリガーになった Change（仮に `X`）** は、
  **トリガーとしてのみ使う**。`X` のラベル・特徴・存在（頭数）は、予測・特徴計算・順位の正解構成に **一切使わない**。
- ある計測時点 `T` で順位づけ・評価の対象になるのは、**その分割（学習なら学習・評価なら評価）に属し、
  かつ `T` 時点で Open な Change だけ**。母集団は **分割内で閉じる**（＝ `Open ∧ 同一分割`）。
- **順位の正解の作り方**: 正解順位は **対象分割の Change だけで詰めて数える**。分割外の Change（`X` 等）が
  間に居ても飛ばして繰り上げる（例: 対象分割で本来 3 位なら、分割外 Change を挟んでも 3 位のまま）。

### 2.5 ビン `b_1 < ... < b_n`（時系列の区切り）
- **ビン幅は「当該リリースを `BIN_COUNT`(=6) 等分」して決める**。すなわち
  `bin_width = (cycle_end − cycle_start) / BIN_COUNT`（例: 当該リリースが 6 か月なら 1 ビン＝1 か月）。
  既定は `BINNING="equal_time"`（当該リリースの時間等分）。各レコード (Change, T) は自分の `T` が入るビンに属する。
- **同じ `bin_width` を前リリース側へも遡って適用**する（前リリースを独自に等分し直すのではなく、当該リリース由来の
  幅をそのまま延長）。これにより学習側ビンが前リリースに及んでよい（§6.1）。
- ビン `i`（時間窓）を「学習期間」、ビン `j`（`j>i`）を「評価期間」として使う（§6・§7）。
- **距離（滞留期間）の最大は `BIN_COUNT` ビン**（縦軸は距離 `d = 1〜BIN_COUNT` の 6 行＝ビン単位）。したがって最深の学習ビンは
  `cycle_start` から **`BIN_COUNT` ビン分＝当該リリース長と同じ期間**だけ遡る。
  → **学習候補プールは `[cycle_start − (cycle_end − cycle_start), cycle_end]`**（＝当該リリース長ぶん前へ）。
  例: 当該 6 か月・前リリース 7 か月なら、前リリースから使うのは直近 6 か月（**最古の 1 か月分だけ少ない**）。
- 検出対象（位置 `p`）は **当該リリース内のビン**に閉じる（学習が前リリースに及ぶのは可。§7.3 とは別物）。
- 同一 Change が学習側ビンと評価側ビンの両方に計測点を持つ場合は **層2**（§2.6）として排他振り分けし、
  学習・評価をまたぐカンニングを防ぐ。

### 2.6 層 1 / 2 / 3（学習・評価の漏れない分け方）
ある (学習期間 = ビン i, 評価期間 = ビン j) を決めたとき、対象 Change を Change の生存期間で 3 層に分ける。
- **層1**: 学習期間内に決着（マージ/放棄）した Change（学習期間だけに存在）。
- **層2**: 学習期間と評価期間の **両方にまたがって Open** な Change（境界をまたぐ）。
- **層3**: 評価期間に入って初めて Open になった（学習期間には居なかった）Change。
- 学習に使うのは **層1 + 層2**、評価に使うのは **層2 + 層3**。
- **重複するのは層2だけ** なので、層2 を Change 単位で **学習用 / 評価用へ排他にランダム振り分け**して、
  同一 Change が学習と評価の両方に出る（＝カンニング）のを防ぐ。
- **振り分け後に実際に使うのは、割り当てた側の期間の T のレコードだけ**:
  - 学習側に割り当てた Change → **学習期間（ビン i）内の T のレコードのみ**を学習に使う。
    その Change の **評価期間（ビン j）内の T のレコードは使わない（捨てる）**。
  - 評価側に割り当てた Change → **評価期間（ビン j）内の T のレコードのみ**を評価に使う。
    学習期間（ビン i）内の T のレコードは捨てる。
  - こうして「一部の計測点だけ学習・残りを評価」にも、「学習側に未来（ビン j）スナップショットが混ざる」
    （時間リーク）にもならないようにする。
- **データの作り方（8:2 と固定数 ＝ いずれも「Change の数」で数える）**:
  - 組み上がる **学習データ全体（層1＋層2学習側） : 評価データ全体（層3＋層2評価側） ＝ 8 : 2**
    （`TRAIN_EVAL_RATIO`、**Change 数の比**）。**8:2 は「層2 をどう割るか」ではなく、最終データ全体の比**。
  - **学習・評価に使う Change 数はどのセル (i,j) でも同じ固定数** `N_TRAIN` / `N_EVAL`（`N_TRAIN:N_EVAL=8:2`）。
  - 各側の内部比は母集団比を保つ（学習内の 層1:層2、評価内の 層3:層2 を各期間母集団の比のまま）。
    **層2 をどちらへ何 Change 振るかは、この固定数と内部比から従属的に決まる**（8:2 で割るのではない）。
  - 各反復（§6.2）で固定数ぶんの Change を新たにサンプリングする（毎回別の部分集合）。
  - 固定数を供給できないセル（層1＋層2 の Change 数 < N_TRAIN、または 層3＋層2 < N_EVAL）は欠測（NaN）。
- **なぜ Change 数で数えるのか（レコード単位にしない理由）**:
  - 同一 Change は T により複数レコードになる/1 レコードのこともある。とくに **長期間 Open だった Change は
    計測点（レコード）を多く持つ**。
  - もし 8:2・固定数を **レコード単位**で取ると、こうした **長寿 Change が少数でデータの大半を占めてしまい**、
    実質的に「ごく少数の Change を学習・評価しているだけ」になりかねない。レコード数を厳密にそろえようと
    すると、ある Change の計測点を途中で切る（推移を壊す）ことになり **バイアス源**にもなる。
  - そこで **数える単位を Change にする**。各セルが同数の「異なる Change」で構成され、Change レベルの
    多様性をセル間でそろえられる。
- **その結果、レコード数（行数）の不一致は許容する（補正しない）**:
  - 固定するのは Change 数なので、実際に学習・評価へ入る **レコード数（行数）はセルや分割で一致しない**
    （8:2 にもならない）。これは上記の理由から **意図的に許容**する。
  - 本質的には「ほぼ差分のない同一 Change のスナップショットを複数回学習するか 1 回学習するか」の違いに
    すぎない（§3.5 の「重みなし複数行」の立場と整合）。

> 層 4/5/6 は **事前分析②（SHAP）** で使う対応概念（評価期間をさらに二段分割したもの）。本書では扱わない。

### 2.7 距離 `d` と位置 `p`
- セル (i, j)（`i<j`）について、**距離 `d = j - i`（＝滞留期間。学習からどれだけ時間的に離れた先を予測するか。
  正方形ビューの縦軸、ビン単位 `d = 1〜BIN_COUNT`）**、**位置 `p = j`（どの時点を予測するか。正方形ビューの横軸）** と定義する。
- 距離・滞留期間は **ビン単位**で測る（1 ビン ＝ リリース長 / `BIN_COUNT`）。実時間に直すとリリースにより
  数か月規模になりうるが、**月固定ではない**（軸ラベルは「距離（ビン）」を基本とし、月数で固定表記しない）。
- 「判断基準が変わった」かどうかは **位置 `p` の違い** で見たい。
  一方、予測精度は当然 **距離 `d` が延びるほど落ちる**（時間が経てば当たりにくい）。
  → §7 のとおり **距離 `d` を固定して位置 `p` を横断比較** することで、距離効果を打ち消し位置依存だけを残す。

### 2.8 評価指標（順位／回帰精度／分類の 3 系統）
モデルが直接予測するのは `time_to_next_review`（時間）。評価は **役割の異なる 3 系統**で多面的に見る:
- **順位**（§2.8.1）: 計測点 T 内でのレビュー優先順位の並びの質。
- **回帰精度**（§2.8.2）: そもそも特徴量が「時間」を捉えているか（信号の有無）。
- **分類**（§2.8.3）: 細かい時刻ではなく「すぐ／放置」という粗い区分を当てられるか。

> 順位 MAE/RMSE は全ペアの順序ずれを等価に罰するため、**差が小さく意味の薄い near-tie の取り違えまで減点**する。
> これを補うため、大きさベースの回帰精度（near-tie を自然に軽くする）と、人間基準の閾値で粗く測る分類を併設する。
> いずれも **モデルの予測値からの後処理**で、追加の学習は不要（同一の予測を 3 系統が共有する。§8 で保存した
> predictions から `recompute` で再計算できる）。

#### 2.8.1 順位ベース（MAE / RMSE / NDCG）
**評価で見るのは順位の質**。
- 予測時間から導出した順位 = 評価集合の Change を **予測時間の昇順**に並べた順位
  （時間が短い＝早くレビューされる＝優先度が高い、を 1 位とする）。
- 正解順位 = 同じ集合を **実 `time_to_next_review` の昇順**に並べた順位。
- **順位は [0,1] に正規化してから指標を計算する（サイズ非依存化）**: 計測点 T のアクティブ集合サイズ `n_T` は
  T ごとに違うので、生の順位（1〜`n_T`）のままだと MAE/RMSE のスケールが `n_T` に依存し、T 間・セル間で
  比較できない。そこで `norm_rank = (rank − 1) / (n_T − 1)`（0〜1）に直してから計算する。
- 指標（各 T で算出する値。MAE/RMSE は正規化順位ベース）:

| 指標 | 定義（各 T、`n_T` 件のアクティブ集合内） |
|---|---|
| `MAE` | `mean(\|予測の norm_rank − 正解の norm_rank\|)`（0〜1） |
| `RMSE` | `sqrt(mean((予測の norm_rank − 正解の norm_rank)^2))`（0〜1） |
| `NDCG@n` | 上位ほど重みを置いてランキングの良さを 0–1 で測る標準 NDCG（上位 `n` 件）。**定義上すでに正規化済み**。relevance は下記で定義 |

> これで 3 指標とも 0〜1 のサイズ非依存スケールになり、T 方向の中央値（§5.7）でまとめても、§7 で位置 `p` を
> 横断比較しても、アクティブ集合サイズの差が紛れ込まない。`n_T < 2` の T は順位が自明なので除外（NDCG@n を
> 意味あるものにするなら最小件数の床を別途設ける。実装ノブ）。

**NDCG の relevance（関連度）= 正解順位ベース**:
- 各 Change の relevance（本当の重要度）は、**実 `time_to_next_review` の昇順（早いほど上位）に並べた
  正解順位を反転した値**とする（一番早くレビューされた Change が最高点）。
- これにより 3 指標すべてが **時間の単位に依存しない「順位ベース」で一貫**する（案 A の `1/(1+時間)` は
  時間単位に値が左右されるため採らない）。
- NDCG は「上位の当て方」を重く見る指標で、MAE/RMSE（全順位のズレを等重みで見る）と相補的。
  実運用でレビュアーが見るのはリスト上位なので、上位重視の NDCG@n を併記する意味がある。
- **`n` の決め方（`NDCG_AT`）**: 「評価集合の中で、次の計測点（翌日）までに実際にレビューされる上位件数」に合わせる。
  - 注意: ランキング対象は実キュー全部ではなく、**固定サンプルした評価集合**（`N_EVAL=100`、うち1日に Open なのは ~60件。§2.6）。
  - 実データ(nova 2015–2019)の全母集団では 1 日あたりレビューされる distinct Change 数の中央値≈67/平均≈65、
    全 Open ~600/日 に対し**レビュー率 ~11%/日**。評価集合 ~60 に同率を当てると **~6件/日**が実際に捌かれる。
  - よって **`NDCG_AT=10`**（評価集合 ~60 の上位 ~16%、実レビュー ~6件に少し余裕）。
    `n=60` は評価集合とほぼ同サイズで識別力ゼロ、`n=6` はノイジーなので、その中間の 10。

> 「予測順位」という語は厳密には「予測時間から導出した順位」を指す略称（認識合わせ済み）。
> 回帰ができれば順位は必然的に出るので、回帰の質と順位の質は別物ではない。

#### 2.8.2 回帰精度（予測時間そのものの誤差・log10 時間で）
順位ではなく、**予測した時間が実時間にどれだけ近いか（＝誤差）**を直接測る系統。裾が極端に長いので
**log10 時間**（`log10(1+hours)`）上で測る（一部の巨大値に支配されないため）。**底 10** にするのは
「差 1 = 10倍ズレ、差 2 = 100倍ズレ」と直感的に読めるため。評価集合の全レコードを **プール** して算出（§2.8.4）。
（学習側の `log1p`＝自然対数とは底が違うが、予測 hours から測り直すので影響しない。`r2_log` は比なので底不変。）

> ここで相関（Pearson 等）は使わない。相関はスケール・ズレに不変で、予測が一律に倍ズレしていても高く出る
> （＝「一緒に動くか＝順序寄り」を見る指標で、精度＝誤差ではない）。順序の質は §2.8.1 が担うので、ここは
> **誤差に徹する**。

| 指標 | 定義（log 時間上） | 意味 |
|---|---|---|
| `mae_log` | `mean(\|log10(1+予測) − log10(1+実測)\|)` | 平均的な誤差。**1 ≈ 10倍ズレ / 2 ≈ 100倍ズレ**の読み |
| `rmse_log` | `sqrt(mean((log10(1+予測) − log10(1+実測))^2))` | 大外しを重く見る誤差（`mae_log` より外れ値に敏感） |
| `r2_log` | `1 − SS_res/SS_tot`（log 時間） | 誤差を正規化（**1=完璧 / 0=常に平均を予測と同等 / 負=平均以下**）。良し悪しを 0–1 で読める |

> 役割: **near-random の原因切り分け**。誤差が大きく `r2_log` がほぼ 0（or 負）なら「順位化以前に時間を
> 当てられていない＝特徴量・データ側の限界」、誤差が小さいのに順位が振るわないなら「順位化の段で落ちている」
> と判断できる。「人間の時間感覚（線形）」の観点は §2.8.3 の分類が担うので、ここは log のまま誤差検出に徹する。
> （絶対スケールの誤差も見たくなったら、上限クリップした線形 MAE を足す余地あり。実装ノブ。）

#### 2.8.3 分類（「すぐ／放置」を粗く当てられるか）
予測時間・実時間を **順序付きバケツ**に離散化して分類として評価する。バケツ内の細かい順序は問わないので、
意味の薄い near-tie を罰しない。**バケツ境界は設定で変更可能**（`constants.CLASS_BUCKETS_HOURS`）とし、
**既定は人間基準の `1日 / 1週間 / 1ヶ月`**（= 24h / 168h / 720h、4 クラス）。評価集合の全レコードを **プール** して算出。

- クラス: `≤1日` / `≤1週` / `≤1ヶ月` / `>1ヶ月`。
- 実データ(nova)での割合 ≈ **9% / 22% / 26% / 43%**（やや「>1ヶ月」寄りの不均衡）。固定境界 1日/1週/1ヶ月は
  分布上およそ 11 / 35 / 60 パーセンタイルに当たる。

| 指標 | 定義 | なぜ使うか |
|---|---|---|
| `macro_f1` | クラスごとの F1 を **単純平均** | 少数クラス（放置）も等しく評価する。**主指標** |
| `micro_f1` | 全クラスの TP/FP/FN を合算した F1（単一ラベル多クラスでは **正解率に一致**） | 「全体の正解率」を併記用に |
| `qwk` | 二次重み付きカッパ。外したバケツの **距離の 2 乗**で罰し、偶然一致を補正（−1〜1） | 順序を踏まえ「だいたい合うか」を評価 |

> 境界の取り方の候補（参考。`CLASS_BUCKETS_HOURS` で差し替え可能）:
> - **固定セマンティック**（既定 1日/1週/1ヶ月）= 解釈・横断比較しやすい。不均衡は `macro_f1`/`qwk` で吸収。
> - **全体四分位を 1 回凍結**（均等クラス。実データでは ~4日/17日/74日）= 均等だが境界の意味は薄い。
> リリースごとに境界を動かすと各リリースの値が別の物差しになり横断比較・drift 判定ができないため、**境界は固定**する。

#### 2.8.4 集約の単位（per-T か pooled か）
セル (d, p) には指標値を **1 個**入れる。多数の (Change, T) レコード（評価ビン内の全 T 分）を 1 値にまとめる
方法が、系統によって違う。

- **順位（2.8.1）**: 順位は「**同じ T で同時に Open な Change の並び**」でしか定義できない（1 件では決まらない）。
  そこで **計測点 T ごとに**順位指標を算出し、**T 方向に中央値**でまとめる（§5.7）。
- **回帰・分類（2.8.2 / 2.8.3）**: 誤差・バケツ分類は **1 レコード単体で計算できる**（同じ T の他 Change を要しない）。
  そこで T で区切らず、**評価ビン内の全レコードをプール（=ひとまとめ）して一度に算出**する。T ごとに割ると
  少数で不安定（特に `r2_log` が定義不能）になるため。

**プールはレコード重み（案1）**:
- 同一 Change は Open な各日 T ごとに 1 レコードになるため、**プールには同一 Change が複数回（重複して）入る**。
  長寿 Change ほどレコード数が多く、その分プール内で **重く効く（レコード重み）**。
- 本書は **案1（レコード重みのまま）** を採用する。実装が単純で、「滞留し続けるレコードも込みで誤差・分類を見る」
  という素直な解釈になる。
- ※ 学習側の重複観測バイアス是正（§2.6 の `CHANGE_BALANCED_WEIGHT`）と揃え、**Change 均等重み
  （各行 1/その Change のレコード数）で集計する案2**も可能（mae_log は重み付き平均、F1 は重み付き集計）。
  まずは案1で分布を見て、必要なら案2に切り替える（実装ノブ）。

> いずれもセル (d, p) の値として §6 の行列に乗せ、§7 で位置 `p` を横断比較する。**3 系統で drift を多面的に確認**し、
> ある系統だけで出た差は他系統で再現しなければ頑健な変化区間とは見なさない。

- 読み込みは既存の共通ローダ **`src.analysis.background_problem.common.data_loader.load_changes`** を再利用

### 3.2 リリース日
- パス: `data/openstack_collected/major_releases_summary.csv`
- `common.data_loader.load_release_dates` / `get_release_cycle` を再利用（サイクル `[cycle_start, cycle_end]` 算出、
  版数と日付が矛盾する異常エントリの除外まで共通実装済み）。

### 3.3 ボット名・人間判定
- priority_distribution と同一の方針（3 一覧の和集合）でボット・自動アカウントを判定し、
  投稿者本人も除いて「人間のレビューコメント」を同定する（§2.3）。
- 当面はそのロジックを本分析の `utils/review_utils.py` に置く（将来 common へ昇格してもよい）。
  一覧: `gerrymanderconfig.ini` の bots / `third_party_ci_accounts.csv` / `extra_bots.txt`。

### 3.4 特徴量
- 既存 `src/features/*` の計算関数を `T` 時点の情報だけで計算して使う
  （`change_metrics` / `bug_metrics` / `developer_metrics` / `project_metrics` / `refactoring_metrics`）。
  本分析側はそれらを呼んで 1 行のベクトルに組み立てるだけ（§5 `features/feature_builder.py`）。
- **`review_metrics.calculate_uncompleted_requests`（未完了リクエスト数）は特徴量から除外する**。
  これにより元の 16 次元ベクトルから 1 つ減り、**本分析の特徴量は 15 次元**となる
  （`review_metrics` は特徴量としては使わない）。
- 同一 Change でも **計測時点 `T` が違えば特徴は変わりうる**（経過時間・リビジョン数等）。
  各レコード (Change, T) ごとに `T` を渡して計算する。
- **高速化（事前集計＋二分探索）**: developer/project 系の 7 特徴（`past_report_count` /
  `recent_report_count` / `merge_rate` / `recent_merge_rate` / `days_to_major_release` /
  `open_ticket_count` / `reviewed_lines_in_period`）は「`T` までの件数」「直近◯日の合計」型の集計なので、
  毎回 3.7万行を走査すると遅い（実測 6.5ms/件 → 日次240万件で約4.3時間）。そこで **全 Change から
  ソート済み配列・累積和を 1 回だけ事前集計**し、各 `T` の問い合わせを **二分探索 O(log n)** で答える
  高速インデックスを用意する（§5.4 `features/fast_index.py`）。1件あたり数マイクロ秒になり日次でも数分。
  - **定義は `src/features` と同一**。「高速版と `src/features` 版が同値」を単体テストで保証する。
  - 安い特徴（`lines_added`/`lines_deleted`/`files_changed`/`elapsed_time`/`revision_count`/
    `test_code_presence`/`bug_fix_confidence`/`refactoring_confidence`）は per-Change で軽いので
    `src/features` をそのまま使う。

### 3.5 学習・評価とも同一 Change の複数行を独立にそのまま使う（クラスタ補正なし）
- 同一 Change の複数レコード（複数計測点）を、**重み付け・間引き・クラスタ集計をせず、そのまま全行**
  学習・評価に入れる。同一 Change が別の `T` で複数回 学習・評価されてよい（推移を素直に扱う）。
- **トレードオフ（意図的な選択として明記）**: 長く Open だった Change ほど行数が多く、学習でも評価でも
  相対的に大きく効く。これは欠陥ではなく立場の帰結（後回しにされがちな PR に焦点を当てる意図と矛盾しない）。
  なお、8:2・固定数を **Change 数で数える**のはこの長寿 Change の影響を避けるため（理由は §2.6）。
- **クラスタ補正はしない**: 「評価で同一 Change の複数行が似て精度を水増しする」懸念に対し、Change 単位
  クラスタ集計やブートストラップでの独立性補正は **行わない**（各 (Change, T) を独立レコードとして数える）。
- 「落とした」代替: Balanced Random Forest 等での自動補正は **ラベルの不均衡**を補正する仕組みで、
  ここで問題になりうる **同一 Change の行数の偏り**とは別物（入れても行数偏りは直らない）。よって採らない。
- **またぎ（同一 Change が学習と評価の両方に出る）だけは禁止**し、§2.6 の層2 Change 単位排他振り分けで防ぐ。
  クラスタを撤廃しても、またぎ禁止によりカンニング（学習で見た Change を評価で当てる）は起きない。

### 5.6 `dataset/splitter.py` … 漏れない学習/評価分け（§2.6）
```python
def split_train_eval(bin_i, bin_j, records, n_train, n_eval, seed) -> (train_records, eval_records):
    """
    学習期間=bin_i, 評価期間=bin_j に対し、各 Change を層1/2/3 に分類し、
    層2 の Change を Change 単位で排他にランダム振り分け（またぎ禁止）。
    train = 層1 + 層2(学習側) の「ビン i 内 T」レコード、eval = 層3 + 層2(評価側) の「ビン j 内 T」レコード。
    - 固定するのは **Change 数**: |train の Change| = n_train, |eval の Change| = n_eval（全セル共通、8:2）。
      選んだ Change の（割り当て側ビンの）**全レコードを入れる**ので、レコード数（行数）は Change ごとに変わり、
      セル間でも一致しない（許容。§2.6）。
    - 各側の内部比（学習内 層1:層2、評価内 層3:層2）は各期間母集団の比のまま保つ。
      層2 をどちらへ何 Change 振るかは n_train/n_eval と内部比から従属的に決める（8:2 で割るのではない）。
    - 供給不足（層1+層2 の Change 数 < n_train など）なら None を返す→セルは NaN。
    """
```
- 層判定は各 Change の `created` と `decision_time`（マージ/放棄時刻）とビン i / j の時間範囲から決める。
- 振り分けは **Change 単位**（またぎ禁止）だが、**使うのは割り当てた側の期間（学習ならビン i、評価なら
  ビン j）内の T のレコードだけ**。反対側の期間の T のレコードは捨てる（§2.6。時間リーク防止）。
- **8:2・固定数はいずれも Change 数で数える。レコード数の不一致は許容**（§2.6）。`n_train`/`n_eval` は
  `N_TRAIN`/`N_EVAL`（None なら全セルの供給可能 Change 数から自動決定。§4）。

### 5.7 `evaluation/ranking_metrics.py`
```python
def rank_mae(y_true, y_pred) -> float: ...   # 正規化順位の MAE（0..1）
def rank_rmse(y_true, y_pred) -> float: ...  # 正規化順位の RMSE（0..1）
def ndcg_at(y_true, y_pred, n) -> float: ... # NDCG@n（0..1、定義上正規化済み）
```
- `y_true` = 実 `time_to_next_review`、`y_pred` = 予測時間。内部で昇順に順位化し、**`(rank−1)/(n−1)` で [0,1] に
  正規化**してから MAE/RMSE を計算（アクティブ集合サイズ `n` に依存させない。§2.8）。`n<2` は除外。
- NDCG の relevance は **正解順位の反転**（実 `time_to_next_review` 昇順の順位を反転、早いほど高 relevance）を
  使う（時間単位非依存・他2指標と一貫。§2.8）。実装コメントで明示する。
- **順位は計測点 T ごとに、その分割・その T のアクティブ集合内で作る**（母集団は分割内で閉じる。§2.4）。
  → **指標は T ごとに 1 つ算出**される（評価ビンに T が 3 回あれば MAE_T が 3 つ）。
- **1 反復内では T 方向に中央値でまとめる**: その反復の代表値 = `median_T(指標_T)`（`T_AGG`、既定 中央値）。
  各 T を等しく扱う（T ごとに Change 数が違っても、その T 内で順位化済みの 1 指標として中央値を取る）。
- **クラスタ補正はしない**（§3.5）。各 T 内の集計では各 (Change, T) を独立レコードとして数える（同一 Change が
  別 T で複数回 評価されてもそのまま）。またぎ禁止（§2.6）でカンニングは起きないため補正は不要。
- 補足: NDCG@n など順位指標が意味を持つには T ごとのアクティブ集合に最低限の件数が要る。極端に小さい T は
  T 方向の中央値で影響が薄まるが、必要なら最小件数未満の T をスキップする（実装ノブ）。

### 5.8 `evaluation/drift_matrix.py` … 学習×予測行列（菱形→正方形）（§6）
```python
def build_matrices(bins, records, metric_names, model_name, n_train, n_eval,
                   n_repeats, t_agg, agg) -> dict[str, np.ndarray]:
    """
    指定 model_name（"lightgbm" / "random_forest"）で構築する。
    n_train/n_eval は全セル共通の固定サイズ（None なら全セルの供給可能数から 8:2 を保てる最大を自動決定。§4）。
    セル (i,j)（i<j, 距離 d=j-i>0）ごとに n_repeats 回繰り返す:
      k 回目は seed=RANDOM_SEED+k で splitter(n_train,n_eval)→regressor(model_name) で 1 回学習・予測し、
      評価ビン内の各計測点 T で順位化して MAE/RMSE/NDCG@n を算出（同一学習から3指標。再学習しない）、
      T 方向に t_agg（既定 中央値）でまとめて「この反復の代表値」とする。
    セル値 = n_repeats 個の代表値を agg（既定 中央値）で集約（2段中央値。§5.7, §6.2）。
    ばらつき(IQR 等)と各反復の代表値(n_repeats 個)も別途返して json へ（SAVE_PER_REPEAT。後で各回を確認できるように）。
    返り値は指標名→行列の dict（例 {"mae": M_mae, "rmse": M_rmse, "ndcg": M_ndcg}）。
    各行列は (学習ビン i, 予測ビン j) では菱形/三角に見えるが、(距離 d, 位置 p=j) に取り直すと欠けのない四角。
    学習側ビン i は前リリースに及んでよい（§6.1）。i>=j（距離 d<=0）は無し。
    評価は T ごとに順位化してクラスタなし独立集計。有効反復が無い／件数不足セルは NaN。
    （実装は (距離 d, 位置 p) で距離固定スライスを取れる内部表現にする、§7）
    """
```
- モデルは指標に依存しないので **1 反復あたり 1 回学習**、そこから 3 指標を計算する（効率のため）。
- **1 セル = n_repeats 反復の中央値**（§6.2）。1 回の精度のブレを抑える。
- **学習 target は `log1p` 変換**（`LABEL_LOG_TRANSFORM`、§4）。`time_to_next_review` は指数的に広がる
  heavy-tail なので、生のまま回帰すると巨大値に損失が引っ張られ中〜下位の予測が雑になる。log で対称化して
  順序を学習しやすくする。**評価は順位ベース＝ log は単調変換で順位不変**なので、指標の定義は変わらず
  「モデルの当てやすさ」だけが上がる（予測も log 空間になるがランキングは順序のみ使うので問題ない）。

## 6. 学習×予測の行列（菱形→正方形に整形した「四角」のイメージ）（§5.8 の中身）

### 6.0 行列の形について（重要）
- 時系列ビンを `n` 個に切り、セルに「ビン i 学習 → ビン j 予測（`i<j`）」の順位ベース指標を入れる。
- これを **(学習ビン i, 予測ビン j) 座標**で描くと、手法.jpg（思想.jpg）のように **菱形（斜めの帯）** に見える。
  だが本質は **軸を取り直すと正方形（四角）**:
  - **縦軸 = 距離 `d = j − i`（＝滞留期間。ビン単位 `d = 1〜BIN_COUNT`。1 ビン＝リリース長/`BIN_COUNT`）**
  - **横軸 = 位置 `p = j`（リリース内のどの時点を予測するか）**
  - これが変化区間.jpg の正方形ビュー。**菱形と正方形は同じデータを別の軸で描いただけ**で、
    `i<j`（過去で学習→未来を予測、距離 `d>0`）の有効領域がそのまま四角に収まる。
    「`i>=j` を NaN にした三角の欠け」は (i,j) 座標で描いたときの見かけにすぎず、(d, p) で見れば欠けのない四角。

### 6.1 前リリースを学習に使ってよい（菱形が境界をまたぐ）
- ビン幅は **当該リリースを `BIN_COUNT` 等分**して決め（§2.5）、その同じ幅を前リリースへ遡って延長する。
- ある (位置 `p`, 距離 `d`) のセルの学習側ビンは `p − d`。`d` が大きい・`p` がリリース序盤だと、
  `p − d` は **前リリースに食い込む**。これは手法.jpg の菱形が V1/V2 境界をまたぐのと同じで、**許容する**。
  判定対象の **位置 `p` は当該リリース内**だが、その学習は前リリース由来でよい。
- 距離（滞留期間）の最大は `BIN_COUNT` ビンなので、最深の学習は `cycle_start` から **当該リリース長ぶん**
  だけ前まで。**例: 当該 6 か月（1 ビン＝1 か月）・前リリース 7 か月 → 前リリースから使うのは直近 6 か月、
  最古の 1 か月分だけ少ない**（前リリースを丸ごとではなく、当該リリース長ぶんだけ遡って使う）。
- 遡り幅は **「リリースの個数」ではなく「当該リリース長（時間）」** で決まる。したがって
  **当該リリースが非常に長いと、遡り幅が前リリース長を超え、前々リリース（さらにその前）にまで及ぶことがある**。
  例: 当該 12 か月・前リリース 4 か月・前々リリース 5 か月 → `cycle_start − 12 か月` まで遡るので、
  前リリースを丸ごと＋前々リリースの一部（8 か月分）を学習に使う。これは許容（自然な帰結）。
- これは §7.3 の「**検出を連結してプールしない**」とは別の話。学習データが前・前々リリースに及ぶのは可、
  複数リリースの行列を 1 枚に連結して位置をまたいで検出するのは不可（検出する位置 `p` は当該リリース内）。

### 6.2 各セルの作り方（1回ではなく反復して中央値）
- 各セルの学習/評価は §2.6 の層1/2/3 分割で漏れなく作る（層2 は Change 単位で排他振り分け＝またぎ禁止）。
  **学習:評価 = 8:2、Change 数は全セル同数の固定数 `N_TRAIN`/`N_EVAL`（レコード数の不一致は許容）**（§2.6）。
  学習・評価とも重みなし複数行・**クラスタ補正なし**（各 (Change, T) を独立レコードに、§3.5）。
- **セル値は 2 段の中央値で作る**（§5.7）:
  1. **T 方向**: 評価ビン内の各計測点 T で順位化して指標_T を出し、その中央値（`T_AGG`）＝この反復の値。
     例: 評価区間に T が 3 回 → MAE_T が 3 つ → その中央値 ＝ 1 反復の代表値。
  2. **反復方向**: それを `N_REPEATS`（既定 10）回繰り返し、各反復の代表値の中央値（`REPEAT_AGG`）＝**セル値**。
- **1 回の学習/評価は不安定**（層2 のランダム振り分け・モデルの乱数で値がブレる）ので、上記 2. の反復で安定化する。
  - `k` 回目は乱数 `RANDOM_SEED + k` を使い、層2 振り分けとモデルを変える。
  - 中央値は外れ値（たまたま悪い分割）に頑健。あわせて **ばらつき（IQR など）も json に残す**（診断用）。
  - **各反復の生スコア（10 回ぶん）も json に残す**（`SAVE_PER_REPEAT=True`）。後から「どの回がどう出たか」を
    確認できるようにする。図（png）には中央値のみを使い、生の反復値は描かない。
- 固定数 `N_TRAIN`/`N_EVAL` を供給できないセルは欠測（NaN）にしてログへ（床 `MIN_TRAIN`/`MIN_EVAL` 未満も除外）。
  `N_TRAIN`/`N_EVAL` が `None` のときは、**含めるセルの供給可能数から 8:2 を保てる最大の固定数を自動決定**し、
  全セルで同数に統一する（決めた値は診断ログ・メタに残す）。
- 四角は「距離 `d` が小さい行ほど当たりやすく、大きい行ほど落ちる」傾向を持つ。
  ここから距離効果を除いて位置依存を見るのが §7。

> 計算量の注意: **モデル学習の回数は「有効セル数 × `N_REPEATS`」だけ**（重いのはここ）。
> 並べ替え検定の `PERMUTATION_N`(=1000) は **計算済みの行列値を並べ替えるだけで再学習しない**ので軽い（§7.2）。
> つまり `PERMUTATION_N` を増やしても学習はやり直さない。学習負荷を下げたいなら `N_REPEATS` を調整する。

## 7. 変化区間の判定方法（主分析の核）

### 7.1 距離を固定して位置を横断比較（raw drop は使わない）
- 変化点は **「各ビンからの精度低下が最大の位置」ではない**。精度の減衰は距離に対して非線形に
  効く（離れるほど加速して落ちる）ため、生の低下量（raw drop）の最大は末尾に出やすく、変化点を取り違える。
- 正しくは **距離 `d = j - i` を固定し、位置 `p = j` を横断比較**する。距離を固定すると距離効果 `f(d)` が
  定数として消え、**位置依存（＝判断基準の変化）だけが残る**。
- 複数の `d` で「同じ位置 `p` が落ちる」ほど、その位置での基準変化が確からしい。

### 7.2 有意性は並べ替え検定で（※モデルは作り直さない）
- 位置間の差が偶然か否かを、並べ替え検定（`PERMUTATION_N` 回）で評価する。`SIGNIFICANCE` 未満を「変化あり」と判定。
- **重要**: 並べ替え検定は **すでに計算済みの行列（セルの精度値）を並べ替えて統計量を測り直すだけ**で、
  **モデルの再学習はしない**。具体的には、距離 `d` を固定した行の中で **位置 `p` のラベル（セル値）を
  シャッフル**して統計量を計算、を `PERMUTATION_N` 回。これは数値の入れ替えだけなので軽い。
- したがって **`PERMUTATION_N=1000` は「数値シャッフルの回数」であって、モデル学習の回数ではない**。
  モデル学習が走るのは行列を埋めるとき（§6.2）だけで、その回数は **有効セル数 × `N_REPEATS`**。
- 帰無分布の作り方（何を交換可能とみなすか）は `drift_detector.py` で固定（既定は上記の「固定距離行で位置を並べ替え」）。

### 7.3 リリース横断は「本数」で集計（プーリングの罠を避ける）
- 変化点の位置はリリースごとに違う（揃わない）。リリースを連結して 1 枚の行列にすると、
  ずれた変化が縦に潰れて検出が壊れ、さらにリリース遷移という別の変動が混ざる。
- したがって **リリースごとに検出し、「N 本中 k 本で変化区間あり」と本数で集計**する（位置は揃えない）。
- 事前分析①の結論はこの集計（変化区間が有意なリリースが十分な割合で存在するか）で述べる。
- 注: ここで禁じるのは **検出を複数リリースで連結・プールすること**。各セルの **学習データが前リリースに
  及ぶこと（§6.1）は別物で、許容**する（検出する位置 `p` は当該リリース内に閉じている）。

## 8. 出力構成

**モデル（lightgbm / random_forest）ごと → さらに指標（mae / rmse / ndcg）ごとにフォルダを分け、png も
指標ごとに出力**する。モデルが最外（§5.11）なので出力も `<project>/<model>/…` で分かれる。

```
data/analysis/preliminary_analysis/concept_drift_existence/
└── <project>/
    └── <model>/                          # lightgbm / random_forest（モデルごと）
        ├── <version>/                    # 例: 20.0.0
        │   ├── mae/                      # 指標ごとにサブフォルダ
        │   │   ├── drift_matrix.csv      # その指標のセル値（反復中央値）（距離 d × 位置 p の四角）
        │   │   ├── drift_matrix.json     # 行列＋セルのばらつき(IQR等)＋各反復の生スコア(10回分)＋メタ情報
        │   │   ├── drift_matrix.png      # 正方形ヒートマップ（縦=距離 d/滞留期間, 横=位置 p, 色=mae）固定スケール
        │   │   ├── drift_matrix_relative.png  # 相対版ヒートマップ（行列内 最良セル=1。コントラスト重視）
        │   │   ├── position_by_distance.csv  # 距離固定の位置別スコア（§7.1）
        │   │   ├── position_by_distance.png  # 距離ごとの位置別折れ線（生の値のみ。検定の印なし）固定スケール
        │   │   ├── position_by_distance_relative.png  # 相対版 折れ線（行列内 最良セル=1）
        │   │   └── drift_test.json       # 検定結果（変化区間の有無・位置・p 値）※png には出さない
        │   ├── rmse/                     # 同上（色・縦軸が rmse）
        │   │   └── …
        │   └── ndcg/                     # 同上（色・縦軸が ndcg@n）
        │       └── …
        └── summary/
            ├── mae/   { drift_count.csv, drift_count.json }   # その指標で N 本中 k 本（§7.3）
            ├── rmse/  { drift_count.csv, drift_count.json }
            └── ndcg/  { drift_count.csv, drift_count.json }
```

- png は各指標フォルダに **4 枚**:
  - 固定スケール版（`drift_matrix.png` / `position_by_distance.png`）… 指標ごとの絶対値域。**バージョン間で比較可能**。
    図には指標名と軸の意味だけを書き「良い向き」は文字では書かない。色は良い/悪いで統一（緑=良い・赤=悪い）（§5.10）。
  - 相対版（`drift_matrix_relative.png` / `position_by_distance_relative.png`）… 「高いほど良い」に揃え
    （MAE/RMSE は `1-値`、NDCG はそのまま）、**行列内の最良セル=1**として正規化（行列内コントラスト重視）。
    各行列を自分の最良で正規化するため **バージョン間比較は不可**、かつ差はノイズを含むので過度に解釈しないこと。
- JSON のメタには project / **model** / version / **metric** / bin 定義 / 各層件数 / 除外（打ち切り）件数 /
  **反復設定（`N_REPEATS` / `REPEAT_AGG`）** / モデル・検定設定を含め、図と結論を再現できるようにする。
  各セルは **中央値（または `REPEAT_AGG`）・ばらつき（IQR 等）・各反復の生スコア（`N_REPEATS` 個）** を持たせ、
  後から各回を確認できるようにする（`SAVE_PER_REPEAT`）。

## 10. エッジケース・注意点
- **タイムゾーン**: Gerrit は naive datetime。比較は naive で統一（既存実装と同じ）。
- **打ち切り（末尾まで未レビュー）**: 既定 `drop`。除外件数はログ・JSON に残す（`survival` 切替の余地）。
- **レコード数不足のセル**: `MIN_TRAIN` / `MIN_EVAL` 未満は NaN にしてスキップ（小標本での不安定な指標を避ける）。
- **ビンが作れない/リリースが短い**: `BIN_COUNT` 個のビンに満たないリリースはスキップしログへ。
- **decision_time の取得**: priority_distribution と同様、`MERGED`/`ABANDONED` の Change は `updated` を
  判断時刻として採用する（終局メッセージは使わない。未決＝`NEW` 等は `decision_time=None`）。
- **NaN を含む行列の検定**: 距離固定の横断比較で NaN は対象から除外（件数はメタに記録）。
- **同一 Change の重複（複数計測点）**: 学習・評価とも重みなしで全行を独立に使用（**クラスタ補正なし**、§3.5）。
  同一 Change が別 T で複数回 学習・評価されてよい。ただし **またぎは禁止**＝層2 は Change 単位で排他振り分けし、
  割り当てた側の期間（学習ならビン i、評価ならビン j）内の T のレコードだけ使い反対側期間の T は捨てる（§2.6）。
- **計測点トリガー X**: 分割外の Change は頭数・順位の分母・正解構成に入れない（§2.4）。
- **再現性**: 層2 のランダム振り分け・モデルの乱数は `RANDOM_SEED` で固定。

   `N_TRAIN`/`N_EVAL`（None なら供給から自動決定、8:2 維持）。各側の内部比は母集団比のまま、層2 の振り分け数は従属的。

   違いにすぎない（§3.5 と整合）（§2.6）。

   ただし **検定の情報（p値・有意マーク・変化点の印）は png 図には描かない**（`drift_test.json` のみ）（§5.10, §8）。
5b. **出力は指標（mae/rmse/ndcg）ごとに分ける**。各指標フォルダに png 2 枚（正方形ヒートマップ＋距離固定の
   位置別折れ線）。セルあたり学習は 1 回で 3 指標をまとめて算出。**良い向きは図に文字では書かないが、
   ヒートマップの色は良い/悪いで統一**（緑=良い・赤=悪い。指標で colormap を反転）（§5.8, §5.10, §8）。

6b. 行列は **「三角」ではなく「四角」**: (学習ビン i, 予測ビン j) で描くと菱形/三角に見えるが、
