# イベント駆動ランキング 設計書（shared_ranking）

## 1. 目的

[src/analysis/trend_models](../trend_models) と [src/analysis/daily_regression](../daily_regression) の両モジュールで利用される「優先順位（rank）」ラベルの算出ロジックを、修論発表資料 `S2520040_kawasaki_20260515.pdf` p.9–11 に示された手法に変更する。

新手法では、

- **説明変数の計測時点**：チケットの作成時点 と リビジョン更新時点（p.10）
- **目的変数（順位）**：各計測時点で未検証のチケットを集合化し、計測時点から検証されるまでの時間が短い順に rank 1, 2, 3, … を付与（p.11）

を採用する。本設計の主要な狙いは以下の 3 点である。

1. 既存の **日次（calendar-day）粒度のスナップショット** を、**イベント駆動（チケットの created / revision_update）粒度のスナップショット** に置き換える。
2. ランキング付けロジックを **共通モジュール `src/analysis/shared_ranking`** に集約し、`trend_models` と `daily_regression` から同一実装・同一キャッシュを参照する。
3. 後段の学習・解析（Learning to Rank / OLS 回帰 / 可視化）は現行 IF を保ったまま新ラベルに切り替えられるようにする。

## 2. 用語

| 用語 | 定義 |
|------|------|
| 計測時点 (measurement_time, T) | あるチケットの **作成イベント** または **リビジョン更新イベント** が発生した瞬間（秒精度） |
| 計測イベント (measurement_event) | 計測時点を生む元イベント。`event_type ∈ {created, revision_update}` と `trigger_change_number`, `trigger_revision_number` を持つ |
| スナップショット (snapshot) | 計測時点 T における「open かつ未検証のチケット集合」 |
| 母集合チケット (population change) | スナップショット内に含まれる個々のチケット |
| 未検証 (unreviewed) | T までの間に「最初の非 bot レビュー」が付いていない状態。`first_non_bot_review_time` が None または `> T` |
| 検証時刻 (verification_time) | 当該チケットに対して T 以降に最初に付いた非 bot レビューの時刻。なければ打ち切り |
| time_to_verify | `verification_time - T`（秒）。打ち切り上限は 1 年（31,536,000 秒） |
| review_priority_rank | スナップショット内で `time_to_verify` を昇順 dense rank した値（1 が最優先） |
| query_id | 1 スナップショットを 1 クエリとして識別するキー |

## 5. ランキング算出アルゴリズム

### 5.1 計測イベントの抽出

各 `change` から計測イベントを以下のように抽出する。

1. `created` 時刻 → `event_type = "created"`, `trigger_revision_number = 1`
2. `revisions` 配下の各リビジョンの `created` 時刻 → `event_type = "revision_update"`, `trigger_revision_number = <revision._number>`（`_number == 1` は created と重複するので除外）

### 5.2 スナップショット母集合の決定

計測時点 T に対して、次の条件すべてを満たす change を母集合に含める。

1. **T 時点で open**：`change.created <= T` かつ
   - `status == "NEW"`、または
   - `status == "MERGED"` で `submitted/merged > T`、または
   - `status == "ABANDONED"` で `updated > T`
2. **T 時点で未検証**：当該 change の messages の中に、T より前の「非 bot かつ owner 以外」のレビューコメントが **存在しない**
3. **解析リリース期間内**：T が `[release_date, next_release_date)` の範囲にある

> 条件 1 は現行 [_is_change_open_at](../trend_models/ranking/daily_rank_builder.py:47) を秒精度に拡張したもの。条件 2 が新規追加（旧手法は「open かつ作成済み」のみで未検証は問わなかった）。

### 5.3 母集合チケットごとの目的変数

母集合の各チケット c について：

- `verification_time(c, T)` = c の messages 中で T 以降の最初の非 bot レビュー時刻（owner 自身 / `tag` が `autogenerated:` で始まるもの / bot は除外。判定ロジックは現行 [_get_first_non_bot_review_time](../trend_models/ranking/daily_rank_builder.py:84) を流用）
- 取得できなければ `verification_time = T + MAX_CENSORING_SECONDS`（=1 年で打ち切り）として `censored = 1`
- `time_to_review_seconds = (verification_time - T).total_seconds()`（負値は 0 にクランプ）
- `reviewed = 1` if 検証あり else `0`

### 5.4 順位付け

`query_id = f"{project}_{release}_{measurement_time:%Y%m%dT%H%M%S}_{trigger_change_number}_{trigger_revision_number}"`

- 1 スナップショット内で `time_to_review_seconds` を昇順に dense rank
- `review_priority_rank_pct = (rank - 1) / max(query_size - 1, 1)`
- `query_size < MIN_QUERY_SIZE`（既定 2）のスナップショットは除外

### 5.5 母集合チケットの特徴量

各 (T, c) ペアに対し、現行 `FeatureExtractor` を用いて c の **T 時点における 16 特徴量** を算出する。母集合は計測時点ごとに再計算するため、計測時点 ≠ そのチケット自身の created/revision の場合でも、その瞬間でのスナップショット特徴量となる。

> 計算量削減のため、特徴量計算は **同一 (change_number, measurement_time) のキャッシュ** を内部で持つ（`Dict[(int, datetime), Dict[str, float]]`）。

## 7. キャッシュ設計

### 7.1 配置

```
data/analysis/shared_ranking/
└── <project>/
    └── <release>/
        ├── event_ranking_<param_hash>.pkl   ← メイン中間データ
        └── event_ranking_<param_hash>.meta.json
```

`<param_hash>` = `sha1` の先頭 16 hex（入力パラメータ + 16 特徴量名 + ロジックバージョン文字列）。

### 7.2 メタ JSON 内容

```json
{
  "project": "nova",
  "release": "19.0.0",
  "period_start": "2018-08-30T00:00:00",
  "period_end": "2019-04-10T00:00:00",
  "min_query_size": 2,
  "max_censoring_seconds": 31536000,
  "feature_names": ["bug_fix_confidence", "..."],
  "logic_version": "shared_ranking@1",
  "row_count": 12345,
  "query_count": 678,
  "generated_at": "2026-05-02T12:00:00"
}
```

`logic_version` を上げると過去キャッシュは無視され再計算される（手法を将来再変更したときの互換切り）。

### 7.3 trend_models / daily_regression 双方からの参照

両モジュールは **同一の `param_hash` を生成できるよう** 共通の `constants.py` を import する。
具体的には以下を共有定数として `shared_ranking.constants` に置く。

- `MIN_QUERY_SIZE`（既定 2）
- `MAX_CENSORING_SECONDS`（既定 31,536,000）
- `FEATURE_NAMES`（16 特徴量。`trend_models` / `daily_regression` の既存定数からはこれを再 export する）

> 既存の `RANKING_MIN_QUERY_SIZE`, `RANKING_MAX_CENSORING_SECONDS`, `MIN_SAMPLES`, `MAX_CENSORING_SECONDS`, `METRIC_COLUMNS`, `FEATURE_NAMES` は `shared_ranking.constants` を参照する形に統一する（破壊的変更ではなく re-export）。

## 11. リスクと検討事項

1. **計算量爆発**：イベント数 ≈ Σ(チケットあたりリビジョン数) になる。Nova 規模では数十万〜百万件オーダー。`feature_evaluator` のメモ化、母集合の絞り込み、`numpy` ベース化が必要になる可能性あり。
2. **未検証判定の境界**：「T までに非 bot レビューが 1 件もない」という定義の厳密な実装が必要。現行 `_get_first_non_bot_review_time` は after_time 以降を見ているため、"before T" の集合に流用するときの実装ミスに注意。
3. **同時刻イベントの衝突**：複数チケットの created/revision_update が同一秒に並ぶケースがある。`query_id` に `trigger_change_number` を含めることで分離する（§5.4）。
4. **打ち切りの一貫性**：両モジュールで `MAX_CENSORING_SECONDS = 31_536_000` で揃える。`shared_ranking.constants` に集約。
5. **後方互換**：戻り値カラム名は維持するが、行数・rank 値は変わる。OLS 係数の絶対値も変動するため、既存の比較レポート・図はすべて再生成扱い。
6. **テスト戦略**：`tests/` 配下に小さな合成 change リスト（3〜5 ticket × 2〜3 revision）を作り、想定 rank が出ることを単体テスト化する。

## 12. オープン項目（要確認）

1. **計測時点の "リビジョン更新" の解釈**：Gerrit における新規 patchset push を指すと理解。draft / WIP の patchset を含めるかどうかは要確認。
2. **未検証判定**：レビューコメントに加えて Code-Review +/-1/+2 のラベル付与も「検証あり」とみなすか。現行は messages のみ見ているため、当面は同じ扱いとする。
3. **解析期間の境界**：チケット created がリリース日より前でも、計測時点（revision_update）が当該 release 期間内ならスナップショットに含める想定で良いか。
4. **複数プロジェクト対応**：現在は nova のみ。`shared_ranking` のキャッシュキーに project を含めて将来拡張可能としているが、初期実装は nova 限定で OK か。
